Serverless Event-Driven AWS Well-Architected

Abstract

This project demonstrates a modern, fully serverless architecture for processing customer orders in an asynchronous manner. By decoupling the submission from the processing logic, we ensure fault tolerance, scalability, and cost-efficiency.

  • Decoupling via Amazon SQS
  • HTTP 202 "Accepted" Logic
  • Dead Letter Queues for Reliability

Core Objectives

Implement Synchronous vs Asynchronous flow for optimal user experience.

Demonstrate Idempotency to prevent duplicate order charges.

Maintain Free-Tier eligibility while using enterprise-grade services.

Service Configuration

API Gateway

Front-door Entry Point

Configured as an HTTP API for lower latency. Direct integration with SubmitOrder Lambda with CORS enabled.

AWS Lambda

Compute Logic

Two functions: Submit (Enrichment/Validation) and Process (Fulfillment/Persistence).

Amazon SQS

The Decoupler

Standard Queue with a 60s visibility timeout. Redrive policy connected to a DLQ after 3 failed attempts.

DynamoDB

NoSQL Storage

Partition Key: orderId. On-Demand capacity to minimize cost while scaling instantly.

CloudWatch

Observability

Centralized JSON logs. Alarms set for ProcessOrder Errors > 0 for proactive notification.

Amazon SES

Notification

Sends transactional order confirmation emails to verified identities in Sandbox mode.

Engineering Excellence Patterns

Load Leveling

SQS buffers messages, allowing the Process Lambda to scale at its own pace without crashing database connections.

Idempotent Consumer

ProcessOrder uses the unique orderId to ensure that even if a message is retried, no duplicate record is created.

Poison Message Handling

Messages that fail 3 times are moved to a Dead Letter Queue, preventing infinite loops and stuck queues.

Least-Privilege Security

Custom IAM roles for each Lambda function grant only the specific permissions needed (e.g., sqs:SendMessage only).
